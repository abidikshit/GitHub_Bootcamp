What is Git?

Key Features:

  • Distributed: Every user has a complete copy of the repository, enabling offline work and decentralized collaboration.
  • Version Control: Tracks changes to files over time, allowing users to revert to previous versions if needed.
  • Branching and Merging: Enables developers to work on separate features or fixes in parallel branches and merge them back into the main codebase.
  • Collaboration:Facilitates team collaboration through features like pull requests, code reviews, and conflict resolution.

How do we use Git?

  • For a new project, the initial step is to inform git that we intend to track files. 
    git init
  • Now, check the files in the folder that need to be committed. 
    git status

When you run git status, Git will show you:

  • Current branch you are on.
  • Staged changes (ready for commit).
  • Unstaged changes (tracked but not staged).
  • Untracked files (not tracked by Git).

Let's run git commit now

Committing a file is like saving a snapshot of the current code in Git's history.

  1. Add the files to our staging area for committing.
    To add all files: 
    git add .
    To add specific files: 
    git add github_bootcamp.html

  2. Now, let's write our commit message to explain the changes. 
    git commit -m "Title" -m "Description"

What happens when we git commit?

  • The staged changes are saved to the Git repository.
  • A commit message describing the changes is added.

What is git branch?

  • In Git, a branch represents a unique set of code changes identified by a distinct name.
  • The default branch when starting a project is typically named main or master.
  • Branching allows developers to experiment with new features or fixes without impacting the main codebase.
  • You can create a new branch, implement changes within it, and later choose to merge those changes back into the primary branch.

How does git branch work?

The git branch command is used to manage branches in Git. Here's how it works:

  1. View Branches:
    To see a list of all branches in your repository: 
    git branch
  2. Create a Branch:
    To create a new branch: 
    git branch new-feature
  3. Switch to a Branch:
    To switch to an existing branch: 
    git checkout branch-name
  4. Create a Branch:
    To create a new branch: 
    git branch new-feature
    Alternatively, you can use: 
    git switch branch-name
  1. Create and Switch to a New Branch:
    To create and switch to a new branch in one step: 
    git checkout -b new-feature
    Or using: 
    git switch -c new-feature
  2. Merge Changes:
    Once you've made changes in a branch and want to merge those changes back into the main branch: 
    git checkout main
    git merge branch-name
  3. Pushing a Branch to a Remote Repository:
    To push a new branch to a remote repository: 
    git push -u origin new-feature
  4. Delete a Branch:
    To delete a branch (only if it has been merged): 
    git branch -d branch-name
    To forcefully delete a branch (even if it has unmerged changes): 
    git branch -D branch-name

Branching Strategies

A "branching strategy" outlines how a software team manages code writing, merging, and deployment using a version control system like Git. It aims to facilitate efficient collaboration among developers sharing the same codebase, ensuring concurrent development without conflicts.


  • Trunks

    If an individual typically works on features on their own, they can utilize their dedicated "trunk" branch. This branch will hold the code related to the feature they are currently developing. Such an approach is suitable for those who tend to work on features one after the other.


  • Feature Branches

    Every new feature has its individual branch! Developers can alternate between branches to handle various features. Even several individuals can collaborate on a single feature by using the same branch.

Command Line Usage

We have changes we made locally that we want to push to GitHub so others can review it.

  • Let's push to GitHub
    • git push <remote> <branch>
  • Typing that URL each time is going to get annoying, let's add it as a remote alias
    • git remote add <name> <url>

Someone made changes to the repository, let's pull down those changes so we can view it locally.

  • The person who m ade the changes needs to push it to GitHub
  • We can pull down the changes to our local machine
    • git pull <remote> <branch>
